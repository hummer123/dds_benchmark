# bcli - Benchmark Client

## 1. 概述

### 1.1 目标

本工具用于评测 **任意基于 DDS/Pub-Sub 模型的中间件** 在目标设备上的性能表现。

**命名由来**：`bcli` = Benchmark Client，用于评测中间件通信性能。

**技术栈**：C++17

核心能力：
- 测量消息吞吐量和带宽
- 测量端到端延迟（含 P50/P90/P99 统计）
- 测量丢包率和可靠性
- 测量 CPU/内存开销

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **简洁输出** | 每行一个指标，机器可解析 + 人类可读 |
| **零配置优先** | 默认配置即可运行，参数最小化 |
| **可重现** | 相同参数下结果可重复 |
| **跨中间件** | 抽象传输层接口，适配任意中间件 |

### 1.3 性能目标

- 发布路径：单核 100% CPU 时，延迟 < 5μs（不含网络）
- 订阅路径：1M msg/s 接收时 CPU < 10%
- 内存占用：< 50MB（不含测试数据）

## 2. 命令行接口

### 2.1 语法

```
bcli <command> [options]

Commands:
  pub <rate> <size>    发布模式
  sub                  订阅模式
  ping [rate]          延迟测试-发送方
  pong                 延迟测试-接收方

Global Options:
  -i <id>      Domain ID (默认: 0)
  -t <sec>    运行时间 (默认: 0=无限)
  -n <N>      Key 实例数量 (默认: 1)
  -b          Best-effort 模式 (默认: reliable)
  -c          开启 CPU/内存 监控输出
  -h          显示帮助
```

### 2.2 命令详解

#### 2.2.1 pub - 发布模式

```
bcli pub <rate> <size>

参数:
  rate    发送频率 (Hz), 0 = 尽可能快
  size    Payload 大小
          数值可选: 64, 256, 1k, 4k, 32k
          单位: B(字节), k(KB), m(MB)

示例:
  bcli pub 0 1k       # 全速发送 1KB 样本
  bcli pub 1000 256   # 1kHz 发送 256B 样本
  bcli pub 100 32k     # 100Hz 发送 32KB 样本
```

#### 2.2.2 sub - 订阅模式

```
bcli sub

订阅并统计接收数据，输出吞吐量、丢包、延迟。

示例:
  bcli sub            # 订阅默认配置
  bcli sub -t 60      # 订阅 60 秒后退出
  bcli sub -c         # 订阅并开启 CPU/内存 监控
```

#### 2.2.3 ping - 延迟测试（发送方）

```
bcli ping [rate]

参数:
  rate    Ping 频率 (Hz), 0 = 尽可能快 (默认: 0)

行为:
  发送 Ping，等待 Pong 响应，计算 RTT 并输出延迟统计。
  Ping 与 Pong 成对工作，需配对使用。

示例:
  bcli ping           # 最快速率测试
  bcli ping 1000      # 1kHz 测试
  bcli ping -c        # 开启 CPU/内存 监控
```

#### 2.2.4 pong - 延迟测试（接收方）

```
bcli pong

行为:
  收到 Ping 立即回复 Pong。
  被动等待，无需参数。

示例:
  bcli pong           # 启动响应者
```

### 2.3 全局选项

| 选项 | 说明 | 默认值 |
|------|------|--------|
| `-i <id>` | DDS Domain ID | 0 |
| `-t <sec>` | 运行时间（秒），0=无限 | 0 |
| `-n <N>` | Key 实例数量 | 1 |
| `-b` | Best-effort 模式 | reliable |
| `-c` | 开启 CPU/内存 监控 | 关闭 |
| `-h` | 显示帮助 | - |

### 3.1 设计原则

- **每秒输出一次**（或每收到数据时）
- **纯文本输出**，便于重定向到文件
- **机器可解析**：固定格式，无彩色输出
- **人类可读**：数值带单位，关键数据居中

### 3.4 延迟百分位解释

延迟百分位是评估通信稳定性的关键指标：

#### 什么是 P50/P90/P99？

将所有延迟测量值**从小到大排序**，取对应位置的值：

| 百分位 | 定义 | 含义 |
|--------|------|------|
| **P50** | 第 50% 的值 | **中位数**，一半的请求比这快 |
| **P90** | 第 90% 的值 | **九成满意**，10% 的请求比这慢 |
| **P99** | 第 99% 的值 | **极端情况**，1% 的请求比这慢 |
| **MX** | 最大值 | 最差情况 |

#### 举例说明

假设测量了 100 次延迟（单位：微秒）：

```
延迟样本 (从小到大): 45, 46, 47, 48, 49, 50, 50, 51, 51, 51, ... 54, 55, 120
```

| 百分位 | 计算方式 | 结果 | 含义 |
|--------|---------|------|------|
| P50 | 第 50 个值 | **50μs** | 一半的请求在 50μs 内完成 |
| P90 | 第 90 个值 | **55μs** | 90% 的请求在 55μs 内完成，10% 超过 |
| P99 | 第 99 个值 | **120μs** | 99% 的请求在 120μs 内完成，1% 超过 |

#### 为什么需要多个百分位？

| 场景 | 看哪个指标 | 原因 |
|------|----------|------|
| **典型性能** | P50 | 排除极端值，看大多数情况 |
| **用户体验** | P90 | 9 成用户能接受的最差情况 |
| **系统稳定性** | P99 | 检测偶尔的卡顿 |
| **故障排查** | MX | 找最大的问题 |

#### 实际场景对比

```
# 好 - 延迟稳定
bcli ping
[T=10.500] M=52.3us  P50=51  P90=54  P99=68  MX=120  C=45000
# 分析: P50=51μs, P90=54μs, P99=68μs
# P90 - P50 = 3μs，延迟波动小，系统很稳定

# 差 - 延迟不稳定
bcli ping
[T=10.500] M=80.5us  P50=52  P90=70  P99=500  MX=1200  C=45000
# 分析: P50=52μs 看起来正常，但 P90=70μs, P99=500μs
# P99 - P50 = 448μs，延迟抖动极大，系统不稳定
```

## 4. Topic 定义

### 4.1 数据结构

> **注意**: IDL `unsigned long` = 4 字节（不同于 C++ `uint64_t` 的 8 字节）

```
IDL 定义 (bcli_perf.idl):

struct Perf4      { unsigned long seq;                     };  // 4B
struct Perf16     { unsigned long seq; octet data[12];     };  // 16B
struct Perf256    { unsigned long seq; octet data[252];    };  // 256B
struct Perf1K     { unsigned long seq; octet data[1020];  };  // 1KB
struct Perf64K    { unsigned long seq; octet data[65532]; };  // 64KB
struct PerfDynamic{ unsigned long seq; sequence<octet> payload; };
struct PerfTimeseries { unsigned long seq; long long timestamp_ns; double values[8]; octet quality; };
struct PerfBatch  { unsigned long batch_id; unsigned long item_count; sequence<octet> items; };
```

### 4.2 Topic 用途

| Topic | 名称 | 用途 |
|-------|------|------|
| 吞吐量 | `Perf256` | Pub/Sub 测试 (256B) |
| 延迟 | `PerfPingPong` | Ping/Pong 测试 (16B) |

### 4.3 关于 Key 的设计决策

#### 4.3.1 为什么性能测试不需要 Key

| 中间件 | Key 机制 | 性能测试影响 |
|--------|----------|--------------|
| DDS/ROS2 | ✅ 显式 @key | 实例路由开销会污染测量 |
| CyberRT | ❌ Channel 替代 | 无实例概念 |
| ZMQ | ❌ 纯消息管道 | 无 key 概念 |
| Kafka | ✅ message key | 消息分片，非实例 |

**结论**：Key 是业务分发逻辑的产物，不是消息传递本身。性能测试应聚焦于消息传递能力，而非业务分发逻辑。

#### 4.3.2 Key 引入的额外开销

在 DDS 实现中，Key 会导致：

```
有 Key 的路径:
  1. 计算 key hash
  2. 查找/创建实例表项
  3. 更新实例状态
  4. Content-filter 检查（可选）
  → 额外 CPU 开销

无 Key 的路径:
  1. 直接序列化/发送
  → 最小开销
```

#### 4.3.3 无 Key 的 IDL 设计

```cpp
// PerfMinimal - 最小基准 (4B)
// 用途: 测试最小序列化/反序列化开销，建立基准线
struct PerfMinimal {
    uint64_t seq;           // 4 bytes
};

// Perf16 - 固定小包 (16B)
// 用途: 测试小数据包处理开销
struct Perf16 {
    uint64_t seq;
    uint8_t data[12];       // 凑满16字节
};

// Perf256 - 固定中等消息 (256B)
// 用途: 测试典型中等大小消息
struct Perf256 {
    uint64_t seq;
    uint8_t data[248];
};

// Perf1K - 固定大消息 (1KB)
// 用途: 测试大消息分片/MTU
struct Perf1K {
    uint64_t seq;
    uint8_t data[1016];
};

// Perf64K - 固定超大消息 (64KB)
// 用途: 测试大消息边界、内存压力
struct Perf64K {
    uint64_t seq;
    uint8_t data[65528];
};

// PerfDynamic - 动态 Payload (可变长)
// 用途: 测试动态内存分配路径
struct PerfDynamic {
    uint64_t seq;
    sequence<uint8_t> payload;    // 可变长
};

// PerfTimeseries - 时序数据
// 用途: 测试连续数据流、批处理
struct PerfTimeseries {
    uint64_t seq;
    int64_t timestamp_ns;       // 时间戳
    double values[8];           // 8个测量值
    uint8_t quality;            // 数据质量标志
};
```

#### 4.3.4 测试矩阵

```bash
# 最小基准测试
bcli -T PerfMinimal     # 4B - 最小开销基准

# 固定大小测试
bcli -T Perf16          # 16B - 小消息阈值
bcli -T Perf256         # 256B - 中等消息
bcli -T Perf1K          # 1KB - 中大消息
bcli -T Perf64K         # 64KB - 大消息边界

# 动态大小测试
bcli -T PerfDynamic -s 1024   # 动态1KB
bcli -T PerfDynamic -s 32768  # 动态32KB

# 时序数据测试
bcli -T PerfTimeseries
```

#### 4.3.5 与 ddsperf 的对比

| 维度 | ddsperf | bcli | 原因 |
|------|---------|------|------|
| Key 设计 | 有 (@key 字段) | 无 | 中间件无关性 |
| 结构数量 | 13个 | 7个 | 精简覆盖核心场景 |
| 嵌套层级 | 4级 | 0级 | 足够测试复杂度 |
| 动态序列 | 1个 | 1个 | 足够 |
| 极值测试 | 无 | 1个 (64KB) | 边界验证 |
| 领域耦合 | CPU监控 | 无 | 通用化设计 |

## 5. 关键参数矩阵

### 5.1 评估维度

| 维度 | 指标 | 测量方式 | 重要性 |
|------|------|---------|--------|
| **吞吐量** | msg/s, Mbps | Sub 计数 | 必须 |
| **延迟** | P50/P90/P99 | Ping-Pong RTT/2 | 必须 |
| **丢包率** | % | 序列号差值 | 必须 |
| **CPU 占用** | % | 资源监控 | 必须 |
| **内存占用** | MB (RSS) | 资源监控 | 必须 |
| **可靠性** | 1-丢包率 | 长时间测试 | 可选 |

### 5.2 延迟评判标准 (同网段环境)

| P50 延迟 | P90 延迟 | P99 延迟 | 评价 | 适用场景 |
|---------|---------|---------|------|---------|
| <50μs | <60μs | <100μs | 优秀 | 实时控制 (>1kHz) |
| 50-200μs | 60-250μs | 100-500μs | 良好 | 常规控制 (100-1000Hz) |
| 200μs-1ms | 250μs-2ms | 0.5-5ms | 一般 | 低速控制 (<100Hz) |
| >1ms | >2ms | >5ms | 较差 | 非实时场景 |

### 5.3 吞吐量评判标准 (1Gbps 网卡)

| 实测吞吐 | 评级 | 备注 |
|---------|------|------|
| >900 Mbps | 优秀 | 接近线速 |
| 700-900 Mbps | 良好 | 有协议开销 |
| 400-700 Mbps | 一般 | 中间件开销较大 |
| <400 Mbps | 较差 | 需优化配置 |

### 5.4 CPU 占用评判标准

| CPU 使用率 | 评级 | 备注 |
|-----------|------|------|
| <15% | 优秀 | 高效处理 |
| 15-30% | 良好 | 正常开销 |
| 30-50% | 一般 | 较多开销 |
| >50% | 较差 | 效率低或瓶颈 |

### 5.5 内存占用评判标准

| RSS 内存 | 评级 | 备注 |
|----------|------|------|
| <50MB | 优秀 | 高效内存使用 |
| 50-100MB | 良好 | 正常范围 |
| 100-200MB | 一般 | 内存占用较高 |
| >200MB | 较差 | 可能存在泄漏 |

| 吞吐量 | 1 msg/s | 计数器最小单位 |
| 丢包率 | 0.01% | 百分比输出 |
| CPU | 0.1% | 使用率输出 |
| 内存 | 0.1MB | RSS 输出 |
| 时间戳 | 1ms | 输出时间格式 |
